## コーディング規約

### 命名規則
- **コンポーネント**: PascalCase（例: `UserCard.tsx`）
- **関数・変数**: camelCase（例: `fetchUserData`）
- **定数**: UPPER_SNAKE_CASE（例: `MAX_RETRY_COUNT`）
- **型・インターフェース**: PascalCase（例: `UserProfile`）

### コンポーネント設計
- 1ファイル1コンポーネントを基本とする
- Propsの型定義は必ず行う
- ロジックはカスタムフックに切り出す

### コメント
- コメントは日本語で記載する
- 複雑なロジックには必ずコメントを付ける
- TODO / FIXME は必ず担当者と日付を記載する

## データベース設計・運用ルール

### 命名規則
- **テーブル名**: スネークケース・複数形（例: `user_profiles`, `order_items`）
- **カラム名**: スネークケース（例: `created_at`, `user_id`）
- **外部キー**: `{参照テーブル単数形}_id`（例: `user_id`, `product_id`）
- **中間テーブル**: 両テーブル名をアルファベット順に繋ぐ（例: `post_tag`）

### マイグレーション運用
- 既存のマイグレーションファイルは原則編集しない（新しいマイグレーションを作成する）
- カラム追加・変更は必ず新規マイグレーションで対応する
- `down()` メソッド（ロールバック処理）も必ず実装する

### Claudeへの指示（DB作業）
- テーブル設計を依頼された場合は以下をセットで作成すること:
  1. ER図（テキストベース）またはテーブル定義の説明
  2. Laravelマイグレーションファイル（`php artisan make:migration` 相当）
  3. ファクトリ・Seeder（開発用ダミーデータ）
  4. Eloquentモデル（リレーション定義含む）
- 要件が曖昧な場合は、テーブル設計の前に確認事項を列挙すること
- 正規化レベル（第3正規形を基本）とパフォーマンスのトレードオフを説明すること

## 環境変数

Gitには含めない。フロント・バックそれぞれで管理する。

**フロントエンド** (`.env.local`)
```
VITE_API_BASE_URL=     # LaravelバックエンドのURL
VITE_APP_TITLE=        # アプリ名
```

**バックエンド** (`.env`)
```
APP_NAME=              # アプリ名
APP_URL=               # バックエンドURL

DB_CONNECTION=mysql
DB_HOST=               # DBホスト
DB_PORT=3306
DB_DATABASE=           # DB名
DB_USERNAME=           # DBユーザー
DB_PASSWORD=           # DBパスワード
```

## Claudeへの作業指示

### 基本方針
- **回答言語**: 日本語で回答すること
- **コメント**: 日本語で記載すること
- **説明レベル**: 専門用語は使ってよいが、背景や意図も合わせて説明すること

### 作業スタイル
- コード変更前に、変更内容と影響範囲を説明してから実装すること
- 複数の実装方法がある場合は、選択肢とトレードオフを提示すること
- テストコードも合わせて提案すること
- 破壊的な変更（ファイル削除・大規模リファクタなど）は事前に確認を取ること

### 自律作業の範囲
- ✅ 自律でやってよい: 新規ファイル作成、コード追加、バグ修正、リファクタ提案
- ⚠️ 確認が必要: 既存コンポーネントの大幅な変更、依存パッケージの追加
- ❌ 勝手にやらない: 環境変数の変更、設定ファイルの変更、本番関連の操作
